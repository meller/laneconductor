# Spec: Track 1062 — Deployment Context & lc deploy Command

## Problem Statement

LaneConductor gives AI agents full context about what the code does (`product.md`, `tech-stack.md`) and how it's developed (`workflow.md`), but has no concept of **how it gets deployed**. This means:
- AI agents generate code without knowing the target runtime (Cloud Run vs Lambda vs Vercel)
- No standard way to trigger deployments from the CLI
- Deployment scripts are ad-hoc per-project (deploy.sh, Makefiles)
- Secrets and credentials end up hardcoded or in .env files that get committed
- No AI-guided deployment setup — users figure it out from scratch every time

## Core Concept

Deployment becomes a first-class citizen alongside code development:

```
conductor/
├── tech-stack.md        ← what it's built with (existing)
├── deployment-stack.md  ← how and where it deploys (NEW)
└── tracks/              ← development work (existing)
```

`deployment-stack.md` is a **context file** — read by AI agents during `implement` and `review` so they can generate deployment-aware code, Terraform, CI config, and environment variable handling automatically.

## Requirements

### REQ-1: `lc setup-deploy` — Guided Deployment Setup

Interactive CLI wizard that:
1. Asks which **deployment template** to use (see templates below)
2. Asks environment questions specific to that template
3. Sets up the deployment infrastructure (creates Terraform files, Firebase config, etc.)
4. Writes `conductor/deployment-stack.md` with the choices made
5. Writes `deploy.json` (machine-readable, like `workflow.json`) for `lc deploy` to use
6. Updates `.gitignore` to exclude secrets
7. Sets up secret management (ADC, Secret Manager, etc.) — **never writes secrets to files**

### REQ-2: Deployment Templates

Six templates covering the most common stacks:

| # | Template | Services | Best for |
|---|----------|----------|----------|
| 1 | **Firebase Full** | Firebase Hosting + Firebase Functions | JAMstack, Next.js static, simple APIs |
| 2 | **GCP Cloud Run** | Cloud Run + Artifact Registry + Cloud SQL (optional) | Containerized apps, FastAPI, Node |
| 3 | **AWS Lambda** | Lambda + API Gateway + S3 + CloudFront | Serverless, event-driven |
| 4 | **Vercel** | Vercel (frontend) + optional backend | Next.js, React, fullstack JS |
| 5 | **Supabase** | Supabase (DB + Auth + Edge Functions) | Postgres-first, auth-included apps |
| 6 | **GCP Full Stack** | Cloud Run + Firebase Hosting + Cloud SQL + Secret Manager | Full production GCP setup |

Each template generates:
- Terraform files (or Firebase/Vercel config) in `infra/`
- Environment variable stubs (no values, just keys) in `.env.example`
- A `deploy.json` with deploy commands per environment
- Updates to `conductor/deployment-stack.md`

### REQ-3: `lc deploy [env]` — Deploy Command

```bash
lc deploy              # deploy to default env (usually prod)
lc deploy prod         # explicit prod
lc deploy staging      # staging
lc deploy preview      # preview/PR deploy (Vercel, Firebase preview channels)
```

Reads `deploy.json` and executes the corresponding deploy command for the environment.
Shows progress, captures logs, updates track status if a track is active.

### REQ-4: `deployment-stack.md` — Context File

Human-readable markdown read by AI agents. Generated by `lc setup-deploy` and kept updated.

```markdown
# Deployment Stack

## Provider
GCP (Google Cloud Platform)

## Environments
- prod: europe-west1, project: makrodash
- staging: europe-west1, project: makrodash-staging

## Services
- Frontend: Firebase Hosting (makrodash.web.app)
- Backend: Cloud Run (macrodash-backend, europe-west1)
- Database: Cloud SQL (Postgres 15)
- Registry: Artifact Registry (europe-west1-docker.pkg.dev/makrodash/macrodash)

## Secret Management
GCP Application Default Credentials (ADC) for local dev.
Secret Manager for production secrets. Never in .env or code.

## IaC
Terraform in infra/. State in GCS bucket: makrodash-tf-state.

## Deploy Command
lc deploy prod  →  runs infra/deploy.sh prod
```

### REQ-5: Zero-Secrets-in-Repo Policy

**The golden rule: no secret values ever touch the filesystem in a committed file.**

Secret management strategy per provider:

| Provider | Local Dev | Production |
|----------|-----------|------------|
| GCP | ADC (`gcloud auth application-default login`) | Secret Manager, accessed via ADC |
| AWS | `aws configure` / IAM roles | Secrets Manager / Parameter Store |
| Azure | `az login` / Managed Identity | Key Vault |
| Vercel | Vercel CLI token | Vercel Environment Variables (encrypted) |
| Supabase | Local `.env` (gitignored) | Supabase Vault / env vars in dashboard |

`lc setup-deploy` enforces this:
- Never prompts for secret values
- Instead generates `.env.example` with key names only
- Adds instructions for each key (where to get it, how to set it via ADC/CLI)
- Verifies `.gitignore` covers `.env`, `*.tfvars`, `service-account*.json`, `*-key.json`

### REQ-6: `conductor/deployment-stack.md` added to scaffold

`/laneconductor setup scaffold` should offer to run `lc setup-deploy` after scaffolding, and if declined, create a stub `deployment-stack.md`:

```markdown
# Deployment Stack

## Status
Not configured. Run `lc setup-deploy` to set up deployment.
```

### REQ-7: AI Agent Awareness

When `deployment-stack.md` exists, agents automatically read it during:
- `implement` — so they write deployment-aware code (e.g. use Secret Manager SDK, not `process.env` directly)
- `review` — so they can flag code that leaks secrets or uses wrong env var patterns
- `quality-gate` — so they can verify infra files exist and secrets policy is followed

### REQ-8: `deploy.json` Machine-Readable Config

```json
{
  "template": "gcp-cloud-run",
  "environments": {
    "prod": {
      "command": "bash infra/deploy.sh prod",
      "region": "europe-west1",
      "project": "makrodash"
    },
    "staging": {
      "command": "bash infra/deploy.sh staging",
      "region": "europe-west1",
      "project": "makrodash-staging"
    }
  },
  "secrets": {
    "strategy": "adc+secret-manager",
    "keys": ["DB_PASSWORD", "JWT_SECRET", "STRIPE_KEY"]
  },
  "gitignore_additions": [".env", "*.tfvars", "service-account*.json"]
}
```

## Acceptance Criteria

- [ ] `lc setup-deploy` runs interactively and writes `deployment-stack.md` + `deploy.json`
- [ ] All 6 templates generate correct infra scaffolding
- [ ] `lc deploy [env]` reads `deploy.json` and runs the correct command
- [ ] No template ever writes a secret value to a file
- [ ] `.gitignore` is updated with secrets exclusions after setup
- [ ] `deployment-stack.md` stub created during scaffold if setup-deploy not run
- [ ] AI agents read `deployment-stack.md` during implement/review/quality-gate
- [ ] GCP ADC, AWS credential chain, Azure Managed Identity, Vercel CLI all documented in generated files
- [ ] `lc setup-deploy` detects existing deploy setup (deploy.sh, Makefile targets) and offers to migrate
